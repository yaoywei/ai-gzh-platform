# 原子化内容管线（Atomized Content Pipeline）

> 将行业资料拆解为结构化"原子"存入飞书多维表格，生成正文时按选题组合原子，保证内容有真实干货。

## 核心思路

传统公众号生产的问题：AI 写的内容"泛泛而谈、没什么干货"（用户原话）。

原子化管线的解法：
1. **收集**：把行业公开资料（报告/政策/案例/技术文档）和公司内部资料上传
2. **拆解**：由 Hermes 把资料拆成一条条飞书多维表格记录（原子），每条只讲一个事实/数据点
3. **组装**：生成正文时，根据选题从多维表格中按类型+标签拉取相关原子，组合成有真实数据支撑的文章

## 4 表架构

### 表 1：行业知识原子库（Atom_Public）
存公开资料拆出来的原子：行业趋势、应用场景、技术概念、政策法规、外部案例、职业/培训。

| 字段 | 类型 | 说明 |
|---|---|---|
| 原子ID | auto_number | PUB-0001 格式 |
| 原子类型 | select | 行业趋势/应用场景/技术概念/政策法规/外部案例/职业培训 |
| 原子标题 | text | 一句话概括 |
| 原子正文 | text | 50-150字，一条只讲一个事实 |
| 关键指标 | text | 如"市场规模1761亿元""效率提升20倍" |
| 场景标签 | multi_select | 外立面巡检/光伏运维/电力巡检/禁毒航测/森林防火/应急救援/环境监测/物流配送等 |
| 来源类型 | select | 官方报告/政策文件/新闻报道/厂商案例/技术博客/科普文章 |
| 来源链接 | url | 原文链接 |
| 图片附件 | attachment | 案例配图、示意图 |
| 商用状态 | select | 可直接商用/需改写/仅内部学习 |
| 置信度 | select | 高/中/低 |
| 更新时间 | updated_at | 系统字段 |

## 数据链路

```
Source_Index（来源+商用状态总控）
  → Atom_Public / Atom_Company（两个原子库）
    → Topic_Recipe（选题配方）
      → 正文生成
```

## 实现侧 Gap 分析（2026-07-10）

> 表已建好、数据已入库。下一步是"组装脚本 + 写作 prompt + 路由集成"。

### 字段类型差异（设计 vs 实际）

设计文档标注 `场景标签` 和 `所需原子类型` 为 multi_select，但实际建表时为 **select（单选）**。如果要支持多标签过滤，需用 `lark-cli base +field-update` 改为 multi_select。

### 字段映射（飞书中文 → JSON 英文）

| 飞书字段名 | JSON key | 值格式差异 |
|---|---|---|
| 原子类型 | atom_type | `["应用场景"]` → `"应用场景"`（解包数组） |
| 原子标题 | title | 直接取 |
| 原子正文 | body | 直接取 |
| 场景标签 | scene_tags | select 单值，非数组 |
| 业务线标签（公司表） | business_tags | select 单值 |
| 关键指标 | metrics | 直接取 |
| 商用状态 | commercial_status | `["可直接商用"]` → `"ok"`，`["需改写"]` → `"need_rewrite"` |
| 保密级别（公司表） | confidentiality | 组装时过滤：仅取"公开" |

### 组装脚本核心逻辑

```
选题配方(TOP-ID)
  → 解析"所需原子类型" + "推荐原子数量"
  → 行业原子库: 按类型+场景标签过滤, 商用状态优先ok
  → 公司原子库: 按类型过滤, 保密级别=公开
  → 输出 atoms.json（中/英文 key 均可，需与 writing prompt 对齐）
```

### lark-cli 数据拉取注意

- `+record-list --page-size 200` 返回 200 条时 Meta 显示 `has_more=true`，但实际可能无第二页——用 `--page-token 200` 验证
- `+record-list` 输出是 **markdown 表格**（非 JSON），解析用 `grep "^| rec"` 提取行、按 `|` 分割列
- 列顺序与 `+field-list` 返回顺序一致，不是字段名顺序——解析时需先查 field-list 确认列序

## 实操经验（2026-07-09）

- **建表用 `lark-cli base +base-create`** 一次建 Base + 初始表 + 字段，比分开建高效
- **strict mode = bot 时只能用 `--as bot`**，user 身份会被拒绝
- **batch create 写入 100+ 条记录**：逐条写（每条一个 JSON 文件 + 一次 CLI 调用）比组装大 batch 更稳定，失败可逐条排查
- **select 字段直接用文本值**（如"行业趋势"），不需要 option ID
- **多选字段用数组**（如 `["外立面巡检", "光伏运维"]`）
- **选项值必须精确匹配已有选项名**，否则报 800030005。先 `+field-list` 确认
- **飞书 Base URL 需要权限**：bot 创建的 Base 默认不授权给 user，需手动添加
