# Execution Gate｜公众号平台严格执行门禁

用于每次按 `ai-gzh-platform` 生产公众号内容、配图、HTML、推草稿前的强制检查。

## 0. 总原则

用户指定 `ai-gzh-platform`、指定模型、指定风格或要求正式交付时，必须严格执行配置和流程。不能用本地占位图、替代模型、手写 HTML、截图拼图、Pillow/SVG 临时图冒充正式产物。

**默认完成范围**：用户说“写一篇 / 用 ai-gzh-platform 写一篇 / 生成公众号内容”时，不要只停在正文。除非用户明确说“只要正文/只写文”，默认一次性执行到“推送/发布前停止”：正文 → 配图 → 视觉 QA → 微信 HTML → 图文包 → 验收表。Step 11 飞书资料包、Step 12 公众号推草稿仍按配置和授权执行；没有明确发布授权时必须在推送/发布前停止。

失败要报阻塞，不要假装完成。

## 1. 配置门禁

开始生产前必须检查 skill 目录下 `config.json`。

如果不存在，或 `setup_status != "configured"`，禁止直接生成文章、图片或 HTML。先完成配置。

必须确认字段：

| 字段 | 要求 |
|---|---|
| `brand_name` | 非空 |
| `content_directions` | 至少 2 个方向 |
| `target_audience` | P1/P2/P3 至少可用 |
| `html_template.style_name` | 必须是支持的 style id |
| `image_style.name` | 必须是支持的 image style id |
| `image_api.endpoint` | 正式配图时必须是真实 endpoint |
| `image_api.model` | 正式配图时必须与用户/配置一致 |
| `image_api.key_env` | 环境变量必须存在 |

用户说“按我画像配 / 你帮我选 / 默认推荐”时，可以补齐已知字段，但必须给依据表。不能把不知道伪装成默认。

## 2. 图片模型门禁

正式生成图片前必须实测图片 API。

验证请求必须使用：

- `config.json.image_api.endpoint`
- `config.json.image_api.model`
- `config.json.image_api.key_env` 对应环境变量

通过标准：HTTP 2xx，且响应包含可下载 `url` 或 `b64_json`。

失败处理：

| 情况 | 必须动作 |
|---|---|
| key env 不存在 | 停止，报告缺少哪个 env |
| HTTP 401/403 | 停止，报告 key 与 endpoint/model 不匹配或无权限 |
| HTTP 非 2xx | 停止，报告状态码和非敏感错误摘要 |
| 返回无 url/b64_json | 停止，报告 API 返回结构不符合预期 |
| 模型名被替换或不一致 | 停止，不能继续 |

严禁：

- 用本地 Pillow / SVG / Canvas 生成图冒充正式图
- 用其他模型替代 `config.json.image_api.model`
- 用别的 endpoint 替代 `config.json.image_api.endpoint`
- API 失败后继续生成正式 HTML 包

只有用户明确说“可以先给占位版 / 降级版 / 本地占位图”，才允许生成非正式占位包，并且文件名和回复必须标注 `非正式占位`。

## 3. 内容生产门禁

正式写文必须走 13 步流程，不能只凭经验手写。

最小可验收链路：

1. 配比轮转判断
2. 爆款调研
3. 选题评分
4. 文章原型判断
5. 按 `references/writing-style.md` 写作
6. 四层自检并修到通过
7. 标题打磨，标题 ≤ 30 字节
8. 落脚点检查
9. 按配置生成配图
10. 按配置生成微信 HTML
13. 交付通知

Step 11 飞书资料包、Step 12 推草稿是可选外部动作。未配置或未授权时跳过，但要说明。

## 4. 四层自检门禁

文章交付前必须输出或保存 QA 结果：

| 层级 | 通过标准 |
|---|---|
| L1 | 禁用词 + 禁用标点零命中 |
| L2 | 口语化词组、节奏、反问、情绪标点达标 |
| L3 | 观点有支撑，不是教科书科普 |
| L4 | 像真人写的，不像 AI 输出信息 |

不通过就修改后重检。

## 4.5 Enterprise Windvane 模式门禁

当选题或用户要求命中 `enterprise-windvane`（AI企业应用 / B端 / 甲方 / 决策者 / SaaS / Agent落地 / ROI / 采购 / 供应商风险）时，必须先读取 `references/enterprise-windvane.md`，并在 L1-L4 之后追加风向标结构自检。

通过标准：

| 项 | 标准 |
|---|---|
| 标题钩子 | 有具体企业场景，并包含数字、损失、结果或反常识之一 |
| 第一屏判断 | 前 80 个汉字内给出明确结论 |
| 错因指出 | 明确写出大多数企业、老板或团队错在哪里 |
| 对照表 | 至少 1 张表，且包含错误做法 / 正确做法 / 业务影响 / 验证指标 |
| 证据点 | 至少 3 个来自输入素材、知识库 atoms、报告数字、企业案例或真实截图描述的证据点 |
| 单一 CTA | 全文只有一个主要承接动作，且与正文承诺一致 |
| B端降噪 | 禁止词零命中：暴利、月入万刀、割韭菜、垃圾、躺赚、白嫖 |

失败处理：任一项不通过，停止进入配图、HTML、飞书资料包和公众号草稿步骤，回到正文修改并重新自检。不得用「风格接近」替代结构验收。

## 5. HTML 交付门禁

HTML 生成前必须确认图片已由指定模型生成并通过视觉 QA。

HTML 必须满足：

- 文件真实存在
- 图片引用正确
- 微信可粘贴，全内联样式
- 不引用本地绝对路径作为线上图片源
- **单独交付 HTML 时必须自包含图片**：用 `data:image/...;base64,...` 内嵌，或同时交付 zip 并明确 HTML 依赖 `imgs/` 相对目录；不要只发一个引用 `imgs/*.png` 的 HTML 文件，否则用户在飞书/浏览器单独打开会看不到图
- 如内嵌 base64，文件大小需可接受；若过大，交付 zip + 说明复制方式
- 若用于推草稿，必须按 `push-draft-pitfalls.md` 预处理，避免 45002/45003/45004

## 6. 视觉 QA 门禁

每张生成图必须用视觉检查确认：

- 没有错字、乱码、遮挡
- 符合配置的 `image_style.name`
- 封面尺寸正确
- 正文图尺寸正确
- 中文可读
- 如是小姚风格，小姚必须有方形眼镜、炸毛短发、橙色卫衣、深灰裤子，并承担核心动作

图不合格必须重生成或局部修复。不能把“差不多”当通过。

**不要过度 QA 卡死**：GPT Image 2 生成中文通常可用。视觉 QA 的阻塞标准是明显错字、乱码、关键文字不可读、IP 特征缺失、风格跑偏、影响理解。极轻微标点/装饰瑕疵不应反复重生成拖慢交付；记录在验收表即可。

## 7. 最终交付验收表

每次正式交付必须附验收表：

| 项 | 结果 |
|---|---|
| skill | ai-gzh-platform |
| config | configured / blocked |
| article workflow | 13步已执行 / 哪步阻塞 |
| image model | 实际 model |
| image endpoint | 已实测 2xx / 阻塞 |
| image files | cover + n 张正文图 |
| visual QA | 通过 / 哪张失败 |
| article QA | L1-L4 通过 / 哪层失败 |
| title bytes | x / 30 |
| HTML | 已生成 / 阻塞 |
| optional publish | 飞书/公众号草稿是否执行 |

没有这张表，不准说“完成”。
