# 5 段式结构 Prompt（v4 写作硬约束）

> 这不是把一篇文章拆成五个小标题，而是把读者从「被一个问题卡住」带到「拿走一个动作」。固定骨架是：卡 → 拆 → 工具 → 结果和坑 → 读者能拿走，最后加边界段。v4 的目标字数是 1500-1800 字，手机阅读时每段不超过 3 行。

## 一、结构总表

| 段 | 实测字数 | 作用 | 必须出现 | 常见失败 |
|---|---:|---|---|---|
| 卡 | 50-100 | 让读者在第一屏认出自己的问题 | 具体人、场景、数字或反差 | 写成背景介绍，钩子不尖 |
| 拆 | 300-500 | 把问题拆成 3-5 个可判断的部分 | 错误、步骤或反常识 | 一口气列 7-8 点，读不完 |
| 工具 | 200-300 | 把动作落实到工具、命令和配置 | 可复制的 bash / Python / YAML | 只说「用某工具」，读者不能跑 |
| 结果和坑 | 200-400 | 交代自己怎么跑、哪里翻车 | 3-5 个真实坑和解决路径 | 只晒成功，读者羡慕完就走 |
| 读者能拿走 | 100-200 | 给 3 个清单和 1 个今天能做的动作 | 关键词钩子或执行清单 | 写成鸡汤或泛泛总结 |
| 边界 | 50-100 | 说清方案不能保证什么 | 1-3 个限制 | 不写，读者期待失控 |

总字数和段落行数是硬约束，不足时补真实过程、命令和坑，不用空话凑长度。标题还要过「具体人 / 场景 / 问题 / 数字 / 方法」五问。

## 二、14 条规则

1. 总字数控制在 1500-1800 字，不回到原版 2000+ 或 ≥2500 的默认。
2. 每个正文段落 ≤3 行；一段只说一个动作或判断。
3. 手机阅读控制在 5-7 屏，多用小标题、数字序号和短句。
4. 卡段只能保留一个核心反差，不在开头铺工具背景。
5. 拆段最多拆 3-5 点，每点都要回答「为什么」。
6. 工具段必须有能复制的完整命令，行尾用 `#` 写作用，并给预期输出。
7. 结果和坑段不写空泛「效果很好」，必须写 3-5 个真实翻车。
8. 每个坑包含现象、根因、解决路径；不能只写「后来修好了」。
9. 边界段必须出现，明确方案不能代替人工判断、预算或持续执行。
10. 不写工具测评，只写「发现重复环节 → 搭流程 → 翻车 → 省掉什么」。
11. 标题和正文使用第一人称、中性称呼、口语短句，不用网红腔。
12. 禁用词和 AI 套路全清：赋能、闭环、颠覆式、抓手、底层逻辑、总而言之、综上、公式化反问、三段式加粗总结。
13. AI 浓度目标 ≤20%；保留自己的数据、犹豫、失败和具体修复，不让 AI 替你下结论。
14. 只引用真实原子、飞书表或实测记录；没有来源的百分比、工时和收入直接删掉。

## 三、每一段怎么写

### 1. 卡：先让一个人停下来

可选三种 Hook：

```text
数字 + 反差：v5 1548 字、阅读 100、分享 7、加微 0。不是写挫，是写错了人。
场景 + 问题：昨晚 11 点，小赵在 4 个公众号前卡住，老板让她明天把视频号也接上。
自己 + 金句：我剪完 18 篇、跑通 14 个对标、改完简介，公众号还是 168 粉。
```

卡段不解释方法，不罗列工具。第一屏只回答：谁在什么时刻，遇到了什么数字化的问题。

### 2. 拆：把一个错因拆成 3-5 点

可以选「4 个错误」「4 个步骤」「3 个反常识」或「5 段式 + 工具栈」。每点用「我原来怎么做 → 为什么不行 → 改成什么」写，读者才能把自己的情况代进去。v6.1 的拆段用了四个错误和一个修复：画像太宽、没找对标、把 AI 当省力工具、没有公众号到私域的动线，最后四件事一起改。

### 3. 工具：必须让读者抄走

工具段不写「建议使用」，直接给最小命令。真实范例：

```bash
# 安装并设置次幂 SDK
pip install cimi-sdk
export CIMI_APP_ID="xxx"
export CIMI_APP_SECRET="yyy"

# 搜索 90 天内的 AI 自动化文章
python3 scripts/cimi_sync.py search-articles \
  --keyword "AI 自动化" --days 90 --min-read 1000 \
  --output feishu/对标账号-候选.csv

# 去重并标注对标账号
python3 scripts/cimi_sync.py dedup-and-tag \
  --input feishu/对标账号-候选.csv \
  --output feishu/对标账号-去重.csv
```

命令后必须写输出样例，例如「79 个候选 → 66 个个人 IP → 14 个强匹配」，并说明密钥需要替换。工具段不承诺结果，只交代可重复的动作。

### 4. 结果和坑：把翻车写在台面上

按「现象 → 根因 → 修复 → 教训」写 3-5 条。师傅的真实坑包括：次幂 `check-balance` 返回 404，原 host 失效；token 鉴权没看客户端源码；v3 画像过宽；菜单栏塞满后读者不知道点哪；文章加 V 前没有免费价值。每条都要有可验证的细节，不能把行业传闻写成自己的结果。

### 5. 读者能拿走 + 边界：给动作，不做升华

读者能拿走段给「今天做什么、明天看什么、哪里回数据」三个清单，末尾接 5 步动线的关键词钩子。边界段冷静写限制：推荐识别有 7-14 天，转化率要靠回流数据，5 个栏目也不代表每个都能稳定跑通。

## 四、可直接喂给 AI writer agent 的 Prompt

```text
你是「大姚｜AI 自动化工作流实战」的公众号 writer agent。

任务：根据我提供的真实原子素材，写一篇给月入 1-3 万内容、运营或小团队负责人的文章。
选题：{topic}
栏目：{column}
目标读者正在经历的场景：{scene}
可用素材（只能使用这里的事实、数字、命令和案例）：
{atoms_json}

严格按以下结构输出，不要输出执行日志：
1. 卡：50-100 字。具体人 + 具体场景 + 具体问题 + 数字或反差。
2. 拆：300-500 字。拆 3-5 个错误、步骤或反常识，并写清原因。
3. 工具：200-300 字。给完整 bash / Python / YAML 命令，逐行写 # 注释和输出样例。
4. 结果和坑：200-400 字。写我实际过程，至少 3 个真实坑；每个坑包含现象、根因、修复。
5. 读者能拿走：100-200 字。给 3 个清单和 1 个今天能做的动作，接关键词钩子。
6. 边界：50-100 字。说清方案不能保证什么。

硬约束：总字数 1500-1800；每段 ≤3 行；标题 ≤30 字节、摘要 ≤54 字节；至少 2 个表格、3 个配图标记；只用第一人称和中性称呼；不写工具测评；禁用赋能、闭环、颠覆式、抓手、底层逻辑、总而言之、综上、公式化反问和三段式加粗总结；不虚构百分比、工时、收入、客户或效果。素材不足就删掉断言或改成「需要自行验证」。
```

## 五、v6.1 范文验收对照

- 卡 87 字：168 粉、18 篇、v5 数据和「写错了人」反差。
- 拆约 380 字：4 个错误 + 1 个修复，不超过 5 点。
- 工具约 280 字：完整 cimi 命令、79→66→14 的输出样例。
- 结果和坑约 280 字：30 天目标和 3 个具体坑。
- 读者能拿走约 200 字：前 7 天日程 + 5 步引流动线。
- 边界约 90 字：推荐识别、加 V 转化、栏目持续性都不做保证。

验收时只要缺一项就退回重写，不用继续润色表面语气。
