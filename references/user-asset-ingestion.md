# 用户素材盘点与分配（2026-07-06 实测）

> 当用户在多轮对话中陆续提供截图/表格/凭证/二维码时，**必须**先盘点 → 分类 → 分配位置，再写文章/出 HTML。不能直接照自己的猜测嵌入。

## 触发条件

用户在任意一轮 message 里发送了**1 张以上**图片（截图/PDF/凭证/二维码/资料包封面），**且**这些图片有"被嵌入文章"的潜在用途。

## 流程（4 步，必走完）

### Step 1: 视觉盘点（vision_analyze 必跑）

对每张图用 `vision_analyze` 跑一遍，**输出**：

```markdown
| # | 文件 | 实际内容 | 主题归属 | 是否当前文素材 |
|---|---|---|---|---|
| 1 | img_xxxx.jpg | 飞书机器人 Hermes 对话截图 | 「72%企业转向AI搜索获客」| ❌ 不是当前「5步30分钟」 |
| 2 | img_yyyy.jpg | 推草稿成功日志（1475字+2图+0残留）| 全链路通证 | ✅ 是当前文 |
| 3 | img_zzzz.jpg | 资料包主图（三栏布局）| 「72%企业」| ❌ 是姊妹篇 |
| 4 | img_wwww.jpg | 交付清单（5项✅）| 当前文 | ✅ 是当前文 |
```

**根因警告**：用户给的图**不一定都是当前文章**的素材。可能是姊妹篇的配图、旧文章的凭证、复用的图。**第一步必须先盘点，不能想当然。**

### Step 2: 3 类资产分类

| 分类 | 定义 | 嵌入方式 |
|---|---|---|
| **当前文核心素材** | 直接证明当前文章流程/数据/效果的图 | 嵌进 Step 1-5 对应位置 |
| **姊妹篇/延伸阅读素材** | 证明作者做过/在做的其他文章配图 | 嵌进文末「延伸阅读」区 |
| **无法归类/质量不符** | 截断/模糊/无关 | 退回让用户重发，不嵌入 |

### Step 3: 位置分配（嵌 HTML 前必输出分配表）

```markdown
## 嵌入位置分配（用户素材 → HTML 位置）

| # | 文件 | 位置 | 标签 |
|---|---|---|---|
| 1 | push-log.png | Step 4 末 | 「全链路通」证据 |
| 2 | delivery-list.png | 文末 | 「5项全 ✅」证据 |
| 3 | feishu-package.png | 延伸阅读 | 姊妹篇主图 |
| 4 | feishu-cover.png | 延伸阅读 | 姊妹篇封面 |
| 5 | style-left.png + AI 合成 | Step 3 末 | 风格对比图（用户给左图+AI 重排） |
```

**规则**：

- **当前文核心素材 → Step 配图位**（占位符 [SCREENSHOT:...] 替换为 [USER-IMG:...]）
- **姊妹篇 → 文末"延伸阅读"区**（新开 `<h2>📚 延伸阅读：xxx</h2>` 块）
- **风格对比图 → 优先用 Pillow 重排**（见 `compose_from_text.py`），不用直接嵌用户截图

### Step 4: HTML 嵌入 + 虚线框规则

- **真实素材嵌进 HTML**：用 `[USER-IMG: ...]` 标记位（不是 `[SCREENSHOT: ...]`）
- **真实素材** = **不要虚线框**（它是已就位的图，不存在"待替换"）
- **虚线框仅用于缺失位**：
  - 橙色虚线（#F59E0B + #FFFBEB）= 用户还没提供的真实截图
  - 蓝色虚线（#2563EB + #EFF6FF）= AI 还没生成的配图
- **微信号 CTA 块**（替代二维码）= 蓝色实线边框，#2563EB + #F0F7FF（不是虚线）

## HTML 中三类素材位的视觉区分

| 类型 | 视觉 | 含义 |
|---|---|---|
| `data:image/png` 直接嵌 | 无边框 | 真实素材，已就位 |
| 橙色虚线框 | `border: 2px dashed #F59E0B; background: #FFFBEB` | 待用户补的截图 |
| 蓝色虚线框 | `border: 2px dashed #2563EB; background: #EFF6FF` | 待 AI 生成的配图 |
| 蓝色实线 CTA 块 | `border: 2px solid #2563EB; background: #F0F7FF` | 微信号/CTA 块（不是虚线） |

**反例（2026-07-06 实际犯的错）**：

- 用户给了 7 张图，我只用了 1 张（风格对比合成的 v2）
- 其他 5 张（推草稿日志 / 资料包主图 / 资料包封面 / 飞书聊天 / 交付清单）**没用**——以为它们是"延伸阅读素材"但没明确告诉用户
- 用户质问"我的几张截图 没有都放进去啊"才发现
- 修正后又过度堆叠：把 5 张都塞进 HTML，文末出现 4 张姊妹篇图，密度过大
- 教训：**先盘点 → 分类 → 分配表 → 再嵌入**，每张图都有明确位置

## 用户原话信号 → 流程触发

| 用户原话 | 触发什么 |
|---|---|
| 「这个是 xxx 图片」「这是给我的...图片」| 启动 Step 1 视觉盘点 |
| 「帮我合」「整合一下」| Step 3 分配表，优先 Pillow 重排 |
| 「图片都嵌进 HTML」/「截图都要用上」 | Step 4 嵌入，所有真实素材图全嵌 |
| 「虚线框也加上」/「虚线框蛮好看的」| 所有"待替换/待生成"位用虚线框（见 SKILL.md 顶部"设计铁律"） |
| 「微信号 Yao...」/「不留二维码了」| 文末用蓝色实线 CTA 块（不是二维码框） |
| 「风格对比/对比图」| 优先 Pillow 重排（compose_from_text.py），不直接嵌截图 |
